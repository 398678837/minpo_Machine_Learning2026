## 1. 나이브 베이즈 텍스트 분류기의 문제점

### 1.1 표준 MultinomialNB의 분류 규칙

표준 다항 나이브 베이즈의 분류 규칙은:

$$\hat{c} = \arg\max_c \left[\log P(c) + \sum_{i=1}^{|V|} f_i \log P(w_i | c)\right]$$

여기서:
- P(c) = N_c / N: 클래스 c의 사전확률
- P(w_i | c): 클래스 c에서 단어 w_i의 출현 확률
- f_i: 테스트 문서에서 단어 w_i의 빈도

### 1.2 파라미터 추정 (라플라스 스무딩)

$$P(w_i | c) = \frac{T_{ci} + \alpha}{\sum_{j} T_{cj} + \alpha |V|}$$

- T_{ci}: 클래스 c의 문서들에서 단어 w_i의 총 출현 횟수
- alpha: 스무딩 파라미터 (기본값 1)
- |V|: 어휘 사전 크기

## 2. 다섯 가지 문제와 해결책

### 문제 1: 조건부 독립 가정의 위반

**문제**: 단어들은 서로 독립이 아니다.

$$P(\text{"machine", "learning"} | c) \neq P(\text{"machine"} | c) \cdot P(\text{"learning"} | c)$$

**영향**: 독립 가정에 의해 관련 단어 쌍의 효과가 과대평가된다. "machine"과 "learning"이 함께 등장하면, 두 단어의 증거가 독립적으로 누적되어 과도한 확신(overconfidence)을 유발한다.

**Rennie et al.의 접근**: 직접적으로 독립 가정을 해결하기보다는, 그 영향을 완화하는 간접적 방법을 채택한다 (가중치 정규화 등).

### 문제 2: 클래스 불균형 (Skewed Class Distribution)

**문제**: 클래스별 학습 데이터 수가 불균형한 경우, 다수 클래스의 파라미터가 더 정확하게 추정된다.

예를 들어:
- 클래스 A: 1000개 문서 -> 파라미터 추정이 안정적
- 클래스 B: 50개 문서 -> 파라미터 추정이 불안정

결과적으로 나이브 베이즈는 다수 클래스로 편향된 예측을 하게 된다.

**해결: Complement Naive Bayes (CNB)**

CNB는 클래스 c의 파라미터를 직접 추정하는 대신, **클래스 c가 아닌 모든 데이터(보완 집합)**로 추정한다:

$$\hat{\theta}_{ci} = \log \frac{\alpha_i + \tilde{T}_{ci}}{\alpha + \sum_{i'} \tilde{T}_{ci'}}$$

여기서 tilde{T}_{ci}는 클래스 c를 **제외한** 모든 문서에서 단어 w_i의 총 출현 횟수이다:

$$\tilde{T}_{ci} = \sum_{j: y_j \neq c} f_{ji}$$

분류 규칙:
$$\hat{c} = \arg\min_c \sum_i f_i \hat{\theta}_{ci}$$

**핵심 아이디어**: "이 문서가 클래스 A에 속하는가?"보다 "이 문서가 클래스 A에 속하지 않는다는 증거가 가장 적은 클래스는?"으로 질문을 재정식화한다.

**장점**:
- 소수 클래스 c의 파라미터를 추정할 때, c를 제외한 다수의 데이터를 사용하므로 추정이 안정적
- 클래스 불균형의 영향이 대폭 감소

### 문제 3: 단어 빈도의 비선형성

**문제**: 다항 모델에서 단어 w가 n번 등장하면 그 증거 기여는 n * log P(w|c)이다. 이는 빈도에 대해 선형적이다. 그러나 실제로 단어의 정보 기여는 빈도에 대해 **오목(concave) 함수**에 가깝다: 첫 번째 등장은 많은 정보를, 100번째 등장은 적은 추가 정보를 제공한다.

**해결: TF 변환**

단어 빈도에 로그 변환을 적용한다:

$$f_i' = \log(1 + f_i)$$

이 변환은 고빈도 단어의 과도한 영향을 줄인다.

### 문제 4: IDF의 부재

**문제**: MultinomialNB는 모든 단어를 동등하게 취급한다. 그러나 많은 문서에 공통적으로 등장하는 단어("the", "is")는 분류에 유용하지 않다.

**해결: IDF 가중치**

$$f_i'' = f_i' \cdot \log \frac{N}{df_i}$$

- df_i: 단어 w_i를 포함하는 문서 수
- 희귀한 단어(df_i가 작음)에 높은 가중치, 흔한 단어에 낮은 가중치

### 문제 5: 문서 길이의 영향

**문제**: 긴 문서는 더 많은 단어를 포함하므로, 특정 클래스의 로그 확률이 더 큰 음수가 된다. 이는 문서 길이에 따른 체계적 편향을 야기한다.

**해결: L2 정규화**

문서 벡터를 단위 벡터로 정규화한다:

$$\tilde{f}_i = \frac{f_i''}{\sqrt{\sum_j (f_j'')^2}}$$

이로써 모든 문서가 동일한 "길이"를 가지게 된다.

## 3. Weighted Complement Naive Bayes (WCNB) 전체 알고리즘

모든 개선을 결합한 WCNB 알고리즘:

```
학습 단계:
1. 각 문서의 단어 빈도에 TF 변환: f' = log(1 + f)
2. IDF 가중치 적용: f'' = f' * log(N / df)
3. L2 정규화: f_tilde = f'' / ||f''||_2
4. 각 클래스 c에 대해 보완 파라미터 추정:
   theta_ci = log((alpha + sum_{j:y_j!=c} f_tilde_{ji}) / (alpha + sum_{i'} sum_{j:y_j!=c} f_tilde_{ji'}))
5. 가중치 정규화: w_ci = theta_ci / sum_i |theta_ci|

예측 단계:
1. 테스트 문서에 동일한 TF-IDF-L2 변환 적용
2. 분류: c_hat = argmin_c sum_i f_tilde_i * w_ci
```

## 4. 실험 결과 상세

### 4.1 20 Newsgroups

| 방법 | Micro-F1 | Macro-F1 |
|------|----------|----------|
| MultinomialNB | 0.774 | 0.758 |
| MultinomialNB + TF-IDF | 0.804 | 0.791 |
| CNB | 0.819 | 0.808 |
| WCNB | 0.847 | 0.840 |
| SVM (linear) | 0.864 | 0.856 |

### 4.2 각 개선의 기여도 분석

| 개선 | 추가 정확도 |
|------|-----------|
| 기본 MultinomialNB | baseline |
| + TF 변환 | +1.5% |
| + IDF 가중치 | +1.5% |
| + L2 정규화 | +1.0% |
| + Complement (CNB) | +3.0% |
| + 가중치 정규화 | +0.5% |

CNB(보완 접근)의 기여가 가장 크며, TF-IDF 변환도 중요하다.

## 5. CNB vs. MultinomialNB: 이론적 분석

### 5.1 이진 분류에서의 동등성

2 클래스 문제에서 CNB와 MultinomialNB는 수학적으로 동등하다. 클래스 c의 보완이 유일한 다른 클래스이기 때문이다.

### 5.2 다중 클래스에서의 차이

3개 이상의 클래스에서 CNB의 이점이 나타난다:
- MultinomialNB: 각 클래스의 데이터만으로 파라미터 추정
- CNB: 다른 모든 클래스의 데이터를 풀링하여 추정 -> 더 안정적

## 6. 논문에서 얻는 통찰

1. **문제를 정확히 진단하면 해결책은 단순할 수 있다**: 나이브 베이즈의 5가지 문제를 정확히 식별한 후, 각각에 대한 해결책은 비교적 단순하다 (로그 변환, IDF, 정규화, 보완 접근).

2. **나이브 베이즈의 잠재력**: 적절한 개선을 통해 나이브 베이즈의 성능을 SVM에 근접하는 수준까지 끌어올릴 수 있다. 이는 나이브 베이즈의 계산 효율성을 유지하면서 성능을 크게 개선할 수 있음을 의미한다.

3. **전처리의 중요성**: TF-IDF 변환, 문서 길이 정규화 등의 전처리가 분류기의 알고리즘 자체만큼 (또는 그 이상으로) 중요하다.

4. **보완적 사고의 힘**: "이 데이터가 클래스 A인 증거"를 찾는 대신 "클래스 A가 아닌 증거가 가장 적은 클래스"를 찾는 보완적 접근이 클래스 불균형 문제를 효과적으로 해결한다.

5. **실무적 권장**: 텍스트 분류에서 나이브 베이즈를 사용할 때는 반드시 TF-IDF 변환과 문서 길이 정규화를 적용하고, 다중 클래스 문제에서는 ComplementNB를 우선 고려해야 한다.
