## 2. 데이터 처리 기반: NumPy와 Pandas

### 2.1 NumPy의 핵심 역할

NumPy의 ndarray는 파이썬 ML 생태계의 **공용어(lingua franca)**이다. 거의 모든 ML 라이브러리가 NumPy 배열을 입출력으로 사용한다.

```python
import numpy as np
from sklearn.linear_model import LinearRegression

# scikit-learn은 NumPy 배열을 입력으로 받는다
X = np.array([[1], [2], [3], [4], [5]])
y = np.array([2, 4, 6, 8, 10])

model = LinearRegression()
model.fit(X, y)  # NumPy 배열을 직접 전달
```

### 2.2 Pandas의 데이터 전처리 역할

Pandas는 실제 데이터(CSV, Excel, SQL 등)와 NumPy/scikit-learn 사이의 다리 역할을 한다:

1. **데이터 로딩**: `pd.read_csv()`, `pd.read_sql()`
2. **탐색적 분석**: `describe()`, `info()`, `value_counts()`
3. **전처리**: 결측치 처리, 인코딩, 스케일링
4. **피처 엔지니어링**: 파생 변수 생성, 그룹 집계

### 2.3 대규모 데이터 처리의 확장

| 도구 | 전략 | 특징 |
|------|------|------|
| **Dask** | 분산 계산 | Pandas와 동일한 API, 청크 기반 처리 |
| **Vaex** | 메모리 매핑 | 메모리보다 큰 데이터 처리 가능 |
| **RAPIDS cuDF** | GPU 가속 | GPU 메모리에서 직접 데이터 처리 |
| **Modin** | 투명한 병렬화 | `import modin.pandas as pd`로 교체만 하면 됨 |

## 3. 전통적 기계학습: scikit-learn

### 3.1 설계 철학

scikit-learn의 성공 요인은 **일관된 API 설계**이다:

```python
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import accuracy_score

# 모든 모델이 동일한 인터페이스를 따른다
# 1. fit() - 학습
# 2. predict() - 예측
# 3. score() - 평가

X_train, X_test, y_train, y_test = train_test_split(X, y)

scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # fit + transform
X_test_scaled = scaler.transform(X_test)         # transform만

model = LogisticRegression()
model.fit(X_train_scaled, y_train)               # 학습
predictions = model.predict(X_test_scaled)        # 예측
accuracy = accuracy_score(y_test, predictions)    # 평가
```

### 3.2 파이프라인 패턴

scikit-learn의 Pipeline은 전처리와 모델을 하나의 객체로 묶는다:

```python
from sklearn.pipeline import Pipeline

pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('classifier', LogisticRegression())
])

pipeline.fit(X_train, y_train)      # 전처리 + 학습을 한 번에
pipeline.predict(X_test)            # 전처리 + 예측을 한 번에
```

### 3.3 지원하는 알고리즘 범위

- **지도 학습**: 선형/로지스틱 회귀, SVM, 의사결정나무, 랜덤포레스트, Gradient Boosting, KNN, 나이브 베이즈 등
- **비지도 학습**: K-Means, DBSCAN, PCA, t-SNE, Hierarchical Clustering 등
- **전처리**: StandardScaler, MinMaxScaler, OneHotEncoder, LabelEncoder 등
- **모델 선택**: Cross-validation, Grid/Random Search, Learning Curves 등

## 4. 딥러닝 프레임워크

### 4.1 TensorFlow

- Google이 개발, 프로덕션 배포에 강점
- TF Serving, TF Lite, TF.js 등 배포 도구 풍부
- TensorFlow 2.0부터 Eager execution이 기본
- Keras가 공식 고수준 API로 통합

### 4.2 PyTorch

- Facebook(Meta)이 개발, 연구에 강점
- 동적 계산 그래프(define-by-run)로 디버깅 용이
- Pythonic한 인터페이스로 학습 곡선이 낮음
- 학술 논문에서의 사용 비율이 급격히 증가

### 4.3 비교

| 측면 | TensorFlow | PyTorch |
|------|-----------|---------|
| 계산 그래프 | 정적 -> 동적(2.0) | 동적 |
| 배포 | 강점 (TF Serving 등) | TorchServe |
| 연구 | 사용 감소 추세 | 지배적 |
| 프로덕션 | 지배적 | 증가 추세 |
| 학습 곡선 | 가파름 | 완만함 |

## 5. 최신 트렌드

### 5.1 자동 기계학습 (AutoML)
- **Auto-sklearn**: scikit-learn 기반 자동 모델 선택/하이퍼파라미터 최적화
- **TPOT**: 유전 알고리즘 기반 파이프라인 최적화
- **H2O AutoML**: 분산 AutoML 프레임워크

### 5.2 해석 가능한 ML (Interpretable ML)
- **SHAP**: Shapley 값 기반 피처 중요도
- **LIME**: 로컬 대리 모델을 통한 해석
- **ELI5**: 모델 예측의 설명 생성

### 5.3 MLOps
- **MLflow**: 실험 추적, 모델 관리, 배포
- **Kubeflow**: Kubernetes 기반 ML 워크플로우
- **DVC**: 데이터 버전 관리

### 5.4 GPU 가속 데이터 과학
NVIDIA의 RAPIDS 프로젝트는 전체 데이터 과학 파이프라인을 GPU에서 실행:
- **cuDF**: GPU 기반 Pandas 대체
- **cuML**: GPU 기반 scikit-learn 대체
- **cuGraph**: GPU 기반 그래프 분석

## 6. 파이썬 ML 생태계의 강점과 과제

### 강점
1. **광범위한 라이브러리**: 거의 모든 ML 알고리즘이 파이썬으로 구현됨
2. **활발한 커뮤니티**: 빠른 버그 수정, 지속적인 기능 추가
3. **풍부한 교육 자료**: 교과서, 온라인 강좌, 블로그
4. **통합 생태계**: 데이터 수집부터 배포까지 하나의 언어

### 과제
1. **GIL**: 멀티스레드 성능 제한
2. **프로덕션 성능**: 순수 파이썬의 속도 한계
3. **패키지 의존성**: 라이브러리 간 버전 충돌
4. **재현성**: 환경 설정의 복잡성

## 7. 핵심 인사이트

1. **생태계가 언어를 이긴다**: 파이썬이 ML의 표준이 된 것은 언어 자체의 성능보다 생태계의 풍부함 때문이다.
2. **API 설계의 중요성**: scikit-learn의 일관된 fit/predict/transform 패턴이 ML 라이브러리 설계의 표준이 되었다.
3. **연구와 프로덕션의 수렴**: PyTorch와 TensorFlow가 서로의 장점을 흡수하며 수렴하고 있다.
4. **전처리의 중요성**: 모델 알고리즘만큼이나 데이터 전처리 도구의 발전이 중요하다.
5. **확장성이 미래의 핵심**: Dask, RAPIDS 등 대규모 데이터 처리 도구의 발전이 파이썬 ML의 미래를 결정한다.
